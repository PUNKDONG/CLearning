# Makefile 使用教程（练习版）

这份文档只讲 **怎么用 Makefile**，不提供你的 6 道题答案。

## 1. 先理解 make 在做什么

`make` 做两件事：
1. 看规则（`目标: 依赖`）
2. 如果目标不存在，或依赖比目标新，就执行配套命令

核心价值：
- 自动按依赖顺序构建
- 只重编译改动过的部分（增量构建）

## 2. 最小 Makefile 结构

你会反复看到这三个目标：
- `all`：默认构建
- `run`：运行程序
- `clean`：清理产物

再加一行：
- `.PHONY: all run clean`

作用：告诉 make 这些是“动作名”，不是文件名。

## 3. 常用命令（必须会）

- `make`：执行默认目标（通常是第一个目标）
- `make all`：显式执行 all
- `make run`：执行 run
- `make clean`：清理
- `make -n`：只打印将执行的命令，不真正执行（排查规则很有用）
- `make -B`：强制全部重建

## 4. 编译和链接要分清

在多文件工程里通常分两步：
1. 每个 `.c` 编译成 `.o`
2. 把所有 `.o` 链接成最终可执行文件

你写规则时，建议显式写出：
- `main.o` 依赖 `main.c` 和它包含的头文件
- `math_utils.o` 依赖 `math_utils.c` 和对应头文件

这样改头文件时，make 才知道该重编译哪些 `.o`。

## 5. 增量构建怎么验证

你可以做三个实验：
1. 只改 `main.c`，再 `make`
2. 只改 `math_utils.c`，再 `make`
3. 只改 `math_utils.h`，再 `make`

然后观察：哪些 `.o` 被重新编译，是否符合你的依赖关系预期。

## 6. 变量和模式规则（避免重复）

当规则变多，建议引入变量：
- `CC`
- `CFLAGS`
- `TARGET`
- `SRCS`
- `OBJS`

再用模式规则处理 `.c -> .o`，减少重复写命令。

## 7. Debug / Release 两套构建

常见做法：
- `debug` 使用调试选项（如 `-g -O0`）
- `release` 使用优化选项（如 `-O2`）
- 两个产物不同名，避免互相覆盖

建议加 `print` 目标，打印当前关键变量，方便排查。

## 8. 静态库练习思路

流程是：
1. 把模块 `.o` 打包成 `.a`
2. 最终程序链接这个 `.a`
3. 改库源码后，重新 make 能更新库并重新链接

## 9. 一套稳定练习流程（每题通用）

1. 先按题目创建文件结构
2. 先写最小可跑规则（哪怕很笨）
3. 确认 `make`、`make run`、`make clean` 都能工作
4. 再优化（变量、模式规则、debug/release、库）
5. 每次修改后先用 `make -n` 看命令是否符合预期

## 10. 你最容易踩的坑

1. 规则命令前面必须是 **Tab**，不是空格
2. 依赖没写全，导致改头文件不重编译
3. 把 `clean` 当文件目标，忘了 `.PHONY`
4. 目标名字和实际输出文件不一致
5. `CFLAGS` 写了但规则里没用到
