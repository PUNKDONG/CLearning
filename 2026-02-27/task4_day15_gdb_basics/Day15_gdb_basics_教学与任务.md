# Day15：GDB 基础（教学 + 今日任务）

## 1. 今天目标

今天只做一件事：学会用 `gdb` 定位 C 程序问题，而不是靠猜。

完成后你要能独立做这 4 件事：

1. 用 `-g -O0` 编译可调试程序。
2. 会下断点、单步、查看变量值。
3. 程序崩溃时会看调用栈（`bt`）定位到代码行。
4. 会用 `watch` 追踪“是谁改坏了变量”。

---

## 2. 编译方式（必须）

```bash
gcc -g -O0 -Wall -Wextra -std=c11 xxx.c -o xxx
```

解释：

- `-g`：生成调试符号（没有它，gdb 很难看清变量和行号）。
- `-O0`：关闭优化，避免调试时代码跳来跳去。
- `-Wall -Wextra`：尽量提前发现可疑代码。

---

## 3. GDB 最常用命令（先背这 12 个）

启动：

```bash
gdb ./app
```

运行与断点：

- `break main`：在 `main` 下断点
- `break 42`：在第 42 行下断点
- `run`：开始运行
- `continue`（`c`）：继续运行到下一个断点

单步：

- `next`（`n`）：单步，不进入函数
- `step`（`s`）：单步，进入函数
- `finish`：运行到当前函数返回

看数据：

- `print x`（`p x`）：打印变量
- `info locals`：看当前函数局部变量
- `display x`：每次停下自动显示 `x`

定位问题：

- `backtrace`（`bt`）：看调用栈（崩溃定位必用）
- `frame 0/1/...`：切换栈帧看上下文
- `list`（`l`）：看源码附近行

监视变量变化：

- `watch x`：一旦变量 `x` 被写入就停下

退出：

- `quit`

---

## 4. 调试标准流程（照着做）

1. `gcc -g -O0 ...` 编译。  
2. `gdb ./app` 进入。  
3. 先 `break main`，再 `run`。  
4. 用 `next/step` 跑到可疑位置。  
5. 用 `print` 看关键变量值。  
6. 崩溃就 `bt`，先看 `frame 0`。  
7. 如果变量被莫名改坏，`watch` 它。  
8. 修代码后重新编译再验证。  

---

## 5. 常见坑（今天必须避免）

1. 没加 `-g` 就进 gdb。  
2. 代码改了但忘了重新编译。  
3. 只看报错信息，不看 `bt`。  
4. 崩溃时不切 `frame` 看参数和局部变量。  
5. 把 `next` 和 `step` 混用导致流程看乱。
